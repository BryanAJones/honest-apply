# Profile Pack + Platform Recommender

You turn a freelancer's platform-neutral identity (in `career-profile.md`) into two things:
1. **A Platform Plan** — which platforms to start on, per wedge, with the cost-to-apply tradeoff.
2. **Publishable Profile Pack copy** — the standing profile a buyer sees when they click you,
   rendered into each chosen platform's actual fields.

This step is **upstream of the gig/proposal flow** (you need a profile before proposals make
sense) but **downstream of the interview** (it reads the facts the interview gathered). It is
**re-runnable** — run it again whenever you add a portfolio piece, change a rate, or add a
platform. It writes copy; it does NOT author new facts.

---

## Before you start

1. Read `career-profile.md` from the working directory. If it doesn't exist, or its **Freelance
   Track sections (4-7) are empty**, stop and tell the user to run `job-application-init` and
   choose the "freelance" or "both" track first. Do not invent a profile.
2. Read `references/platform-economics.md` (recommender rules + cost-to-apply + fact-freshness)
   and `references/platform-render-maps.md` (per-platform fields/limits).

**Hard rule — anti-invention:** every value comes from `career-profile.md`. Never guess a rate,
a metric, or a claim. Anything missing becomes a visible `[fill in: ...]` / `[confirm: ...]`
placeholder. The user fills facts; you only arrange and re-cut them.

---

## Step 1 — Platform Recommender (do this first; it decides where the copy goes)

Read the user's profile signals: discipline/wedge, **portfolio strength** (and crucially
whether it's *showable* vs merely *exists/rough* — "showable ≠ exists"), **track record**
(new/no-reviews vs established/JSS), and **rate tier / seniority**.

Apply the recommender rules table in `platform-economics.md`. For EACH wedge, output:

```
[Wedge]
  START HERE → <platform> — <one-line why> (cost to apply: <pay-per-bid / free / vetting-gate>)
  ADD NEXT   → <platform> — <why>
  SKIP/LATER → <platform> — <why not yet>
```

Rules to honor:
- A **build-capable-but-thin-proof** profile (real work, no demos/traction, no reviews) is NOT
  "strong portfolio" — don't send it to portfolio-forward platforms as a spearhead. Lead with
  call+trial platforms (Wellfound) and deliver-to-prove cold-start gigs.
- High-end **strategic/fractional** work skips generalist bid pools.
- Stamp the plan with the **verification date** of the platform facts used; resolve any ⚠️
  unverified facts (a quick web check) before relying on them.

Write this into `career-profile.md` Section 8 (Platform Plan), replacing the
`[generated by job-application-profile]` placeholder.

---

## Step 2 — Render the Profile Pack per platform

Decide structure with the user: **one blended profile** or **per-wedge Specialized Profiles**
(Upwork allows up to 2). Then, for each chosen platform from Step 1's plan, render using its map
in `platform-render-maps.md`:

- Map the neutral source (positioning, overview, services+rates, portfolio projects, skills)
  into the platform's fields, respecting every limit (e.g. Upwork title ≤ ~70 chars; the
  overview's first ~2 lines carry the hook before the "more" fold; ≤ ~15 skill tags).
- Portfolio items: prefer **media (screenshots/video)** over a link when work is login-gated or
  private — a public link is not required on most platforms.

---

## Step 3 — Honesty pass (run before showing anything)

Enforce the same guards as the rest of the kit (see `references/writing-style-defaults.md`):

- **Maturity/traction:** no rendered claim may exceed a Proof Ledger entry's maturity tag.
  A `deployed-no-users` project is never "live / in users' hands / shipped."
- **Solo/team:** team projects claim only the user's slice — never "I built it" / "my product."
- **Verify-before-cite:** any link in the rendered copy must be flagged `[confirm loads well for
  a stranger — not login-walled/empty; current name]` unless already verified.
- **IP:** never recommend exposing a `keep-private` item; render it as describe-only/media.
- **Anti-slop:** apply the banned-phrase + structural-tell checks; for thin-proof profiles,
  ensure the overview ends with the **paid-trial** offer rather than overselling.

---

## Step 4 — Output

Write the rendered profiles to `freelance-profiles.md` in the working directory (one section per
platform / Specialized Profile), each clearly labeled with its source wedge. Include the Platform
Plan at the top. Then:

- Show the user the plan + the rendered copy. Invite edits — they own every word.
- Tell them: copy is ready to paste into each platform; the `[fill in]`/`[confirm]` markers are
  the only things needing their input (real facts you couldn't invent).
- **Never publish, submit, or post.** This step produces copy; the user posts it.

Tell them the next step: "Paste a real gig to get a SPEND/SKIP verdict + a drafted proposal"
(the gig branch of `job-application`).

---

## Re-running

This step is idempotent on facts: re-run it after any profile change and it re-renders cleanly.
It never edits the user's recorded facts in `career-profile.md` Sections 4-7 — only Section 8
(the Platform Plan) and the generated `freelance-profiles.md`.

---

## Standing rules
- Never invent a fact. Missing = visible placeholder.
- Never claim beyond a Proof Ledger entry's honesty fields.
- Never push the user to expose private/commercial IP for portfolio cred.
- Platform facts are dated priors — re-verify per the fact-freshness convention.
- Never publish or post. Tee it up; the user posts.
